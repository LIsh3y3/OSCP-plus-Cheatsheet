# Debugger(=Sources)

- [わかりやすい参考記事](https://ics.media/entry/190517/)

## 基本

- 主にJavaScriptのソースコード閲覧・動作確認に使用
- 変数上にマウスのカーソルを合わせると、変数の値を確認できる
- 現在の変数の値や、処理がどのように進んでいるのかを確認できる
- {}を押すと、読みやすい形式に変換される

##### メリット

- ⭐️：注入したXSSペイロードがどのようなcontextで処理されているかがわかる。
	- XSSペイロードにより元の文法を壊してはいけない

- 注入箇所にブレークポイントを設定し、さらに"Watch"パネルに追加することでcontext(現状の変数の値など）がわかる
![[Pasted image 20240204101614 1.png]]


## ブレークポイントの設定

- ブレークポイントはその部分が処理されたら一旦==処理を止める==
- 該当処理箇所が青くハイライトされる

流れ：
1. 行番号をクリックすることでブレークポイントを設定する
2. ⏸️(Pause script execution)を押し、色がなくなることを確認
3. プログラムを実行するとブレークポイントで止まる

### 主要機能の説明

![[Pasted image 20231227223134.png | 300]]
- リジューム実行：次のブレークポイントまで処理を飛ばす
- ステップイン：行ごとにプログラムを実行していく
- ステップオーバー：大まかな処理を確認する。関数の内部の処理をすべて実行し次の行で止まる
- ステップアウト：現在処理が止まっている関数内の残りの処理をすべて実行し、関数を呼び出している行へ移動
- ➡️・：　ステップ。次の行に

# Inspector(=Elements)

![[Pasted image 20231227220428.png]]
- このボタンを押した後、ページの特定の場所をクリックするとその特定の場所の要素が表示される
- 主な用途は、ブラウザ上の表示や動作のシミュレーションを即時反映で確認すること
	- （実際のファイルに変更が反映されるわけではない）

- hiddenフィールドの探索が可能
	- session keyなどの機密情報が隠されている可能性がある
	- →*Search*タブでも検索可能（検索にはSearchタブが適している）

##  🚨注意

- DOMとして構築された要素の「名前・属性・値・テキストノード」" に対する部分一致検索をするため、HTMLの文字列の検索はできない
	- つまり要素ごとの検索にしか使えない
- HTMLの文字列を検索するのであれば、Ctrl + Shift + Fで開く*Search*タブか、Elementsタブではなく*Sources*タブで検索すること

### Elementsタブでの検索可否の例

| 検索対象                     | ヒットの可否 | 理由                     |
| ------------------------ | ------ | ---------------------- |
| `title`                  | ヒットする  | タグ名が`title`としてDOMに存在   |
| `Sign In`                | ヒットする  | テキストノードにマッチ            |
| `title>Sign In`          | ❌      | DOM構造にはそのような文字列が存在しない  |
| `<title>Sign In</title>` | ❌      | 文字列としてのタグ構造はDOMには存在しない |
| div                      | ヒットする  |                        |
| div id                   | ヒットしない |                        |
