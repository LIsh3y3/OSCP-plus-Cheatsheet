## HTTP requset smuggling(HRS)とは

- 単一のユーザだけでなく、複数のユーザから送られてくるHTTPリクエストのシーケンスがWebサイトによってどのように処理されるかを操作するテクニック
- セキュリティ上の制御をバイパスし、機密データに不正アクセスしたり、他のユーザを侵害できる

- 基本HTTP/1と関係が深い
- HTTP/2でもバックエンドのアーキテクチャによっては脆弱

---
## Request smugglingで起こること

###### フロントエンドのリクエスト処理方法

- 近年、フロントエンドサーバ(LBやリバースプロキシ)がバックエンドサーバにリクエストを振り分ける形式が一般的
- フロントエンドサーバは効率的に処理するために、複数のリクエストを同じバックエンド接続上で一続きに送る
- 受信サーバは、==あるリクエストがどこで終わり、次のリクエストがどこで始まるかを判断==しなければならない
![[Pasted image 20240126160721.png | 400]]

###### 攻撃者はどう悪用するか

- フロントエンドサーバが1つのリクエストとして処理した内容が、バックエンドサーバによって2つ目のリクエストの開始として扱われるようにリクエストを作成する
	- 不完全なリクエストを密輸する
- これは次のリクエストの先頭に追加される。
- アプリケーションがそのリクエストを処理する方法(値の検証など）を妨害する。
![[Pasted image 20240126161035.png | 400]]

## Request smuggling脆弱性の発生原因

- HTTP/1.1では「1つのリクエストの終了」を意味する2つのヘッダをサポートしている
	- `Content-Length` (**CL**)
	- `Transfer-Encoding`(**TE**)

#### CL: Content-Length

- メッセージbody部の長さをバイト数で表す
```http
POST /search HTTP/1.1
Host: normal-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 11

q=smuggling
```

#### TE: Transfer-Encoding: chunked

- HTTPの転送エンコーディングの1つ
- 転送開始時点でデータサイズ(`Content-Length`)がわからない場合に使われる
- データを「チャンク」というブロックに区切って転送する
- チャンクサイズはバイト数を16進数で表し、その次にテキストもしくはバイナリのデータが来る
- 最後にチャンクサイズ0として終了を意味する
```http
POST /search HTTP/1.1
Host: normal-website.com
Content-Type: application/x-www-form-urlencoded
Transfer-Encoding: chunked

b                 //チャンクサイズ
q=smuggling　　　　//チャンクデータ
0　　　　　　　　　　//チャンクサイズが0＝データの終了
```

### CLが複数、もしくはCLとTEの両方を指定することによる解釈の違い

- RFC上ではTEを優先してCLを上書き、もしくはCLを無視するとしている
- 現実のソフトウェアではRFC通りに実装していないケースがある
- CLとTEの両方を含む不正なリクエストで「1つのリクエストの終了」の解釈の違いを引き起こす

###### 問題が起きる2つの理由

- サーバによっては`Transfer-Encoding`ヘッダをサポートしていない
- `Transfer-Encoding`ヘッダをサポートしていても、難読化されていると処理しないことがある
