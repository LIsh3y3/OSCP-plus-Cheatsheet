#### ヘッダパラメタインジェクションに使用するヘッダ

- `alg`ヘッダパラメタのみが必須
- 以下のような攻撃に使えるヘッダがある
- これらのヘッダは、署名を検証する時どの鍵を使うかを受信側のサーバに伝える
	-  [[📕#検証(JWT)]]

- `jwk`
- `jku`
- `kid`

## jwkパラメタを使用した自己署名JWTの注入

###### jwkパラメタとは

- jwk(JSON Web Key)
- 鍵を表す埋め込みJSONオブジェクト
```json
{
    "kid": "ed2Nf8sb-sD6ng0-scs5390g-fFD8sfxG",
    "typ": "JWT",
    "alg": "RS256",
    "jwk": {
        "kty": "RSA",
        "e": "AQAB",
        "kid": "ed2Nf8sb-sD6ng0-scs5390g-fFD8sfxG",
        "n": "yy1wpYmffgXBxhAUJzHHocCuJolwDqql75ZWuCQ_cb33K2vh9m"
    }
}
```

###### 攻撃概要

- サーバが`jwk`パラメタに埋め込まれた鍵をそのまま使ってしまう脆弱性を悪用する
	- 本来はJWT署名検証のための鍵はホワイトリストで指定されるべき

- 修正したJWTに対して、攻撃者自身のRSA秘密鍵を使って署名し、対応する公開鍵を`jwk`ヘッダに埋め込む

### jwk header injectionによるJWT認証バイパスステップ

1. JSON Web Tokenタブでヘッダの`"alg": "RS256"`なことを確認する(公開鍵暗号方式)
2.  JWT Editor -> New RSA Key -> Format: JWK, Key size: そのまま -> Generate
3. JSON Web Tokenタブでペイロードを目的の値に編集する
4. "Attack" -> "Embedded JWK"を選択し、ステップ1で生成したRSA keyを選択する(これでsignまでされる)
5. リクエストをSend

## jkuパラメタを使用した自己署名JWTの注入

###### jkuパラメタとは

- jku = JSON Web Key Set URL
- サーバが正しい鍵を含む鍵セットを取得できるURLを提供する
```json
{
    "keys": [
        {
            "kty": "RSA",
            "e": "AQAB",
            "kid": "75d0ef47-af89-47a9-9061-7c02a610d5ab",
            "n": "o-yy1wpYmffgXBxhAUJzHHocCuJolwDqql75ZWuCQ_cb33K2vh9mk6GPM9gNN4Y_qTVX67WhsN3JvaFYw-fhvsWQ"
        },
        {
            "kty": "RSA",
            "e": "AQAB",
            "kid": "d8fDFo-fS9-faS14a9-ASf99sa-7c1Ad5abA",
            "n": "fc3f-yy1wpYmffgXBxhAUJzHql79gNNQ_cb33HocCuJolwDqmk6GPM4Y_qTVX67WhsN3JvaFYw-dfg6DH-asAScw"
        }
    ]
}
```

###### 攻撃概要

- `jwk`ヘッダで公開鍵を直接埋め込む代わりに、`jku`を使用して鍵を含むJWK SetをURLで参照させる
- 署名を検証する時、サーバは指定されたURLから目的の鍵を取り出す
	- 通常は`/.well-known/jwks.json`エンドポイントに格納されていることがあり取り出せる事がある

- もし鍵を取得するURLにフィルタがかかっているなら、[[2.  一般的なSSRF防御の回避#SSRF ホワイトリスト回避]]のテクニックを試すことでバイパスできる可能性がある

###### JWK Set

- 異なるキーを複数表すJWKの配列を含むJSONオブジェクトのこと
```json
{
    "keys": [
        {
            "kty": "RSA",
            "e": "AQAB",
            "kid": "75d0ef47-af89-47a9-9061-7c02a610d5ab",
            "n": "o-yy1wpYmffgXBxhAUJzHHocCuJolwDqql75ZWuCQ_cb33K2vh9mk6GPM9gNN4Y_qTVX67WhsN3JvaFYw-fhvsWQ"
        },
        {
            "kty": "RSA",
            "e": "AQAB",
            "kid": "d8fDFo-fS9-faS14a9-ASf99sa-7c1Ad5abA",
            "n": "fc3f-yy1wpYmffgXBxhAUJzHql79gNNQ_cb33HocCuJolwDqmk6GPM4Y_qTVX67WhsN3JvaFYw-dfg6DH-asAScw"
        }
    ]
}
```

### jku header injectionによるJWT認証のバイパスステップ

1. JSON Web Tokenタブでヘッダが`"alg": "RS256"`であることを確認する
2. JWT Editor -> New RSA Key -> Format: JWK, Key size: そのまま -> Generate
3. 作成された鍵を右クリック -> "Copy public key as JWK"
4. エクスプロイトサーバ上のファイルに以下をホストする(`Content-Type: text/html`)
```json
{
    "keys": [
       //ここにステップ2でコピーした公開鍵をペースト
    ]
}
```

5. JSON Web Tokenでペイロードを目的の値に編集
6. さらにヘッダを追加編集し、以下の値へ
```json
"jku": "[攻撃者のサーバ上のファイル絶対URL]"
```
7. Sign -> 生成した鍵を選択 -> "Update/generate, `"alg"`, `"typ"`, and `"kid"` parametars"を選択しOK
	- 生成した秘密鍵で署名される

8. リクエストをSend

- もし`jku`パラメタに指定できるURLがフィルタかかっている場合は[[2.  一般的なSSRF防御の回避#SSRF ホワイトリスト回避]]のテクニックが使える

## kidパラメタを使用した自己署名JWTの注入

###### kidパラメタとは

- Key ID
- 複数の鍵から選択する必要がある場合に、サーバが正しい鍵を識別するための IDを提供する。
- トークンに含まれている `kid` から、サーバ側で保持している鍵の集合の中から対応する `kid` を持つ鍵を探す

###### 攻撃概要

- 検証のための鍵は[[#JWK Set]]としてサーバに格納されている
	- [[📕#検証(JWT)]]

- JWS仕様では`kid`の構造は定義されていないので、開発者がDBの特定エントリを指定したり、ファイル名をしていることもある
	- [[1. JWT attackの基本#JWS (JSON Web Signature)]]

- ヘッダの`kid`パラメタが[[Path traversal]]に脆弱な場合、サーバ上の任意のファイルを検証鍵として使わせる事が可能になる
- さらにもし、`"alg": "HS256"`(共通鍵暗号方式)で、かつ、Linuxシステムの場合は`/dev/null`を指定して空の文字列を鍵として使用させ、検証用の鍵も空の文字列になる
```json
{
    "kid": "../../dev/null",
    "typ": "JWT",
    "alg": "HS256",
	...
}
```

- もしサーバが検証用の鍵をDBに保存する場合、`kid`パラメタはSQLiの潜在的なベクターとなる([[1. SQLi基本と対策]])

### kid header path traversalによるJWT認証バイパスステップ

1. JSON Web Tokenタブでヘッダの`"alg": "HS..."`なことを確認する(共通鍵暗号方式)
2. `kid`パラメタにパストラバーサルを使用して`/dev/null`を指定する
```json
"kid": "../../../../dev/null",
```
3. JSON Web Tokenでペイロードを目的の値に編集
4. Attack -> Sign with empty key -> Algorithmで正規のアルゴリズム`HS...`を選択する
5. リクエストをsend

###### cty(Content Type)

- JWTペイロードのメディアタイプを宣言する
- 省略されていてもサポートされている可能性もある

- 上記のテクニックで署名検証をバイパスできた場合。
- ↓[[4. 隠れたXXEi脆弱性の発見#Content-Type編集によるXXE攻撃]]を可能にする
```json
"cty": "text/xml"
```
- ↓[[1. 安全でないデシリアライズの基本と対策]]を可能にする
```json
"cty": "application/x-java-serialized-object"
```
