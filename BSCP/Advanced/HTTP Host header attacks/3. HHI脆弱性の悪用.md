## 認証のバイパス

###### 基本概念

- 一般的に、Webサイトは特定の機能へのアクセスを内部ユーザだけに制限する
- しかし、Webサイトのアクセス制御機能の中には、`Host`ヘッダに簡単な変更を加えるだけで、これらの制限をバイパスできる場合がある。
- これは、他のエクスプロイトへの展開を可能にする

###### 関連

- [[1. Access controlの基本と対策]]

### ステップ

0. [[2. HHI脆弱性の検出]] を念頭に置いて以下を実行していく
1. 高権限者のみアクセス可能なパスを直接指定する。例：Adminパネル
2. "Admin interface only available to local users"というエラーを確認する
3. Hostヘッダをlocalアドレスに変更する
```http
GET /admin HTTP/2
Host: localhost
```

## virtual hostへの総当たりによる社内Webへのアクセス(実践無)

###### 攻撃可能なシチュエーション

- 1つのサーバで仮想的に公開Webサイトと社内Webサイトの両方をホストしている

###### 基本概念

- サーバには通常、公開用のIPアドレスとプライベートなIPアドレスがある
```/etc/hosts
www.example.com: 12.34.56.78
intranet.example.com: 10.0.0.132 //社内Webサイト
```

- プライベートなWebサイトは公開されているDNS情報に載っていないので、DNS情報を見ただけでは社内Webサイトの存在はわからない

###### 攻撃方法概要

- Information disclosure([[3. 一般的な情報開示ソース]])などの手段で社内Webサイトのドメイン名を発見し、直接そのドメイン名をリクエストしてアクセスする
- もしドメイン名を知らなければ、Intruderを使って、候補となるサブドメインをブルートフォースする
	- 同じサーバにホストされていれば、トップレベルドメイン(`com`)とセカンドレベルドメイン(`example`)が同じ可能性が高い

## Routing-based SSRF

- = Host header SSRF attack
- [[3.  Blind SSRF]]
### Routing-based SSRFとは

- 通常のSSRF：　XXE([[2. XXE攻撃によるファイル取得とSSRF攻撃]])や、ユーザ制御可能な入力から生成されたURLに、HTTPリクエストを送信するビジネスロジックの脆弱性に基づいている
- Routing-based SSRF：　中間コンポーネント(LBやリバースプロキシ)を利用することに基づく
	- 中間コンポーネントが内部NW全体へのゲートウェイになる

###### 攻撃概要

- `Host`ヘッダに攻撃者の任意のシステムドメインを設定し、中間コンポーネントにリクエストを転送させる

###### Routing-based SSRF脆弱性の原因

- 中間コンポーネントが`Host`ヘッダの値を検証せずにリクエストを転送するから。
- 中間コンポーネントは特権的なネットワークに属す。
- しかも、外部から直接リクエストを受け付ける。

### ステップ

###### 1. Collaboratorを使用し脆弱性を特定する

- `Host`ヘッダにCollaboratorサーバのドメインを指定する
- ターゲットサーバや他の経路上にあるシステムからのDNS lookupを確認する
- ＝ 任意のドメインにリクエストをルーティングできる可能性がわかる

###### 2. 内部システムのプライベートIPアドレスを入手する

1. リクエストやレスポンスにプライベートIPアドレスが記載されてないか？
2. その企業に属するホスト名をスキャンして、プライベートIPアドレスに解決するものがあるか？
3. `192.168.0.0/16`のようなプライベートIPアドレスのレンジでブルートフォースできないか

## Connection state attack

###### HTTP/1.1の接続再利用について

- リクエストヘッダで`Connection: close`と指定 : 接続を1回限りの利用
- `Connection: keep-alive`と指定：接続を再利用。レスポンスヘッダに`Keep-Alive: timeout=10`と表示される

###### 接続の再利用が悪用される原因

- 脆弱なサーバは、同じ接続を利用するなら、`Host`ヘッダのような特定のプロパティが何度目のリクエストでも、==最初のリクエストと同じ値をもつと仮定==する
- 最初のリクエストの値のみ厳重に検証する場合もある
- この場合、最初のリクエストで無害な値を送り、同じ接続を利用して悪意あるペイロードを送信することができてしまう

###### 他の攻撃への展開の可能性

- 多くのリバースプロキシは`Host`ヘッダを利用してルーティングする。
- もし、リバースプロキシが同じ接続上のリクエストは最初のリクエストでの`Host`の値で固定だと勘違いしているとする
- 攻撃者はこの挙動を利用して、`Host`ヘッダを変えることで意図しないサーバや内部ネットワーク上のシステムにリクエストを送る
- `Host`ヘッダを悪用した攻撃へ...
	- [[#Routing-based SSRF]]
	- [[Password reset poisoning]]
	- [[1. Web cache poisoningの基本と対策]]

### ステップ

1. HTTPリクエストヘッダで`Connection: close` から`Connection: keep-alive`へ変更してリクエスト。
2. 2つのリクエストをRepeater上でGroup化する。
	- 通常のリクエストで、`Connection: keep-alive`としたものを1つ目のリクエスト
	- 目的のアクセス先をリクエスト行と`Host`ヘッダとに指定したものを2つ目のリクエスト
```http
GET / HTTP/1.1
Host: example.com
Connection: keep-alive
```
```http
GET /admin HTTP/1.1
Host: 192.168.0.1
Connection: keep-alive
```

3. "Send group in sequence(single connection)"し、2つ目のリクエストのレスポンスで目的のアクセス先に到達できたか。
4. ステップ3で到達したアクセス先で特定の機能を実行したい場合、ブラウザ上で機能を実行し(失敗する)、HTTP historyからリクエストを入手する
5. 入手したリクエストをステップ2の2つ目のリクエストとし、ステップ3を実行して目的の処理を実行する
```http
GET / HTTP/1.1
Host: example.com
Connection: keep-alive
```
```http
POST /admin/delete HTTP/1.1
Host: 192.168.0.1
...
csrf=token&username=carlos
```

## 不正なリクエスト行によるSSRF(実践無)

- 例えばリバースプロキシがリクエスト行からパスを入手し、`http://example.com`を先頭につけてルーティングする場合
- 攻撃者は前の値をクレデンシャルとして認識する`@`を先頭に付与したリクエストを送る
```http
GET @private-intranet/example HTTP/1.1
```

- リバースプロキシは以下のように変更する
```
http://example.com@private-intranet/example
```

- これにより、攻撃者は`private-intranet`(内部NW)に`example.com`というユーザ名でリクエストが可能になる
