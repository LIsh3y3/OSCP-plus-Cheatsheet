- Not only SQL

- Big Dataとの関係が深い
- SQL以外でもデータを扱える。幅広いクエリ言語を使用する
	- カスタムされたクエリ言語
	- XMLやJSONなど

### 従来の関係モデルとの違い

###### メリット

- 結合操作を利用しない(LEFT JOINなど)
- スキーマが固定されていないので自由
- 非構造化データの取り扱いが可能
- 水平スケーラビリティ(スケールアウト)が確保しやすい。RDBはスケールアウトするのが一般的
- 制約が少ない(自由)

###### デメリット

- 結果整合性のみを保証し、トランザクションのACID属性を保証しない
- 高度なトランザクション処理は利用できない
- データの格納と取得が高度に最適化されている。そのために機能性を最小限にしていることが多い
- 制約が少ない(危険)

### NoSQLの分類

###### Key-value型

- 値が一意のキー文字列に関連づけられている。
- 値はキーに基づいて検索される
- RedisやAmazon DynamoDBなどがある

###### ソート済みカラム指向型
- RDBは行単位でデータを扱うが、カラム指向型は列単位で扱う。
- 1行単位でデータを追加、削除するような処理は苦手だが、データを列単位で扱う集計処理は得意
- Apache CassandraやApache HBaseなどがある

###### ドキュメント指向型

- 柔軟性のある半構造化ドキュメントにデータを格納する。
- JSON、BSON、XMLなど
- APIやクエリ言語でクエリされる。
- XMLやJSONを扱うアプリはデータをそのまま利用・管理できるのがメリット
- *MongoDB*(一番人気）やCouchbaseなどがある。

###### グラフ指向型
- ノード、エッジ、プロパティから構成されるグラフ構造。ネットワーク(つながりの意）を扱うアプリで使われる
- Neo4jやAmazon Neptuneなどがある
