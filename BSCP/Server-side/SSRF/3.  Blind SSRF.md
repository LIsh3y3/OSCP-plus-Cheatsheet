
###### Blind SSRFとは

- WebアプリがバックエンドにHTTPリクエストを送ることができるが、レスポンスがフロントエンドに返ってこない場合のSSRF脆弱性
- 影響は他のSSRFにくらべ低い。情報窃取には使用不可能。
- ただ、*RCE*の脆弱性につながる可能性もある

#### Blind SSRF脆弱性の発見

- OASTを使用：　攻撃者が制御する外部システムへのHTTPリクエストをトリガーし、作用を監視する
- ツールはCollaboratorを使用：ドメイン名を生成し、ペイロードとしてアプリケーションに送信し、それらのドメイ ンとのやりとりを監視する。
	- アプリケーションからの HTTP リクエストがCollaboratorのドメインで観測されれば、SSRF に対して脆弱
###### Tips

- Collaboratorのドメインに対しDNS look-upは観測できるがHTTPリクエストが観測できないことが多い
- これは外部への名前解決は必要な機能として実装されていることが多いが、予期しない宛先のHTTP接続はネットワーク層のフィルタリングでブロックされることが多いから。

#### Blind SSRFエクスプロイト方法

###### サーバ自身や他のバックエンドシステム上の他の脆弱性を探す

- well-knownな脆弱性を検出するために、InternalなIPアドレス空間を総当たりするペイロードを送信できる。
 - ペイロード中にBlind out-of-bandのテクニックも存在する場合、パッチが適用されていない内部サーバのクリティカルな脆弱性を発見できるかも。。。
	-  Blind out-of-band は、外部との通信を発生させて、それが発生したか否かによって間接的に挙動を把握する

###### 攻撃者の制御するシステムに接続したHTTPクライアントに悪意のあるレスポンスを返す

- もしサーバの HTTP 実装における深刻なクライアント側の脆弱性を悪用できれば、アプリケーションのインフラストラクチャ内でリモ ートコードの実行が可能になるかも
