## 基本

- ==Webアプリの文脈では認証とセッション管理が重要==

### アクセス制御の種類

###### DAC, MAC, RBAC(role-based)、Programmatic access control

- Programmatic access controlはユーザの権限などがDBに保存され、プログラムでこの権限を参照するなどする

###### 垂直アクセス制御

- 異なるタイプのユーザが異なるアプリケーション機能にアクセスできる。
- 例えば、adminはどのユーザのアカウントでも変更または削除することができるが、一般ユーザはこれらの操作にアクセスできない

###### 水平アクセス制御

- 異なるユーザーが同じ種類のリソースのサブセットにアクセスできる
- 例えば、銀行アプリでは、あるユーザーは自分の口座から取引を見たり支払いを行ったりすることができるが、他のユーザーの口座にはアクセスできないなど

###### コンテキストによるアクセス制御

- アプリの状態やユーザのインタラクションに基づき、機能やリソースへのアクセスを制御する
- 例えば、小売業のWebサイトでは、ユーザーが支払いを済ませた後にショッピングカートの中身を変更できないようにする

## アクセス制御の脆弱性対策

- [ ] 難読化だけに頼らない
	- アクセス先のページをランダムな文字列にしたり、ユーザにGUIDを割り当てるだけでは不十分

- [ ] 一般に公開するディレクトリ以外はデフォルトでアクセス拒否の設定をする

- [ ] 可能な限り、アクセス制御を強制するためにアプリケーション全体で単一のメカニズムを使用する
	- アプリケーション全体で同じメカニズム（例えば、特定の認証ルーチンやRole-Based Access Control (RBAC)のようなアクセス制御フレームワーク）を使用することで、アクセス制御を一元化する
	- アクセス制御のヌケモレを減らす効果がある
	- 具体的には、認証サービス（例：OAuth、JWTなど）を導入して、その認証サービスを通過したユーザのみがアプリケーションの特定のリソースにアクセスできるようにする

- [ ] コードレベルでは、開発者に対しては各リソースに対する許可されたアクセスを宣言してもらい、デフォルトではアクセスを拒否すること
	- つまり、アクセスを許可するリソースをソースコード中に明示的に記載し、集約する

- [ ] アクセス制御が設計通りか、徹底的に監査とテストを行う

- [ ] 基本的に、高権限の処理は台帳などに実行者、実行日時、用途などを記載し、上長の承認を経てから権限を委譲すること。
