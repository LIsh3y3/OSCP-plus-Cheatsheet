###### Second-order SQLi(= Stored SQLi)とは

- First-order SQLiは普通の。
- 攻撃者のHTTPリクエストをアプリケーション側で将来の使用のために保持しておく。後ほどの攻撃が目的なので最初の時点で何も問題は引き起こされない。
- しかし、後ほど異なるHTTPリクエストを処理するときに、アプリケーションは攻撃者があらかじめアプリケーションに保持させておいたクエリを危険な方法でSQLクエリに組み込み、実行する

## SQLi 脆弱性の検出方法

###### 自動
- Burp Scannerを使用(Pro version)

### 手動

- すべての入力箇所に以下のステップで試していく
	- URLのクエリパラメタ
	- Cookie
	- フォーム...など

1. ⭐️シングルクォート`'`もしくはダブルクォートを使って、エラーやその他の異常(internal Server Errorなど)起こせれば、SQLiの脆弱性アリと判断
	- ユーザの入力をそのままクエリに注入している事が分かるから

2. 正当な値と不適切な値を入力したときのレスポンスに違いがないか調べる
3. `OR 1=1` と `OR 1=2` のような、Boolean条件を設定し、アプリケーションのレスポンスの違いを調べる。
4. SQLクエリ内で実行されると時間遅延を引き起こすように設計されたペイロードで、応答にかかる時間の違いを調べる。(Blind SQLiのTime-based用)
5. OASTペイロードは、SQLクエリ内で実行されたときに帯域外のネットワーク・インタラクションをトリガーし、その結果のインタラクションを監視する

#### クエリのどの部分に脆弱性があるのか

- ほとんどがSELECTクエリのWHERE句だが、SQLiが発生する他の一般的な場所は以下のようになる

- UPDATE文：WHERE句、もしくはアップデートされた値
- INSERT文：挿入された値
- SELECT文：テーブルまたはカラム名
- SELECT文：ORDER BY句

## 機密情報の窃取

- `-- //`を挿入する方法。
	- `-- ` : SQLの行コメント。空白語の値をコメントとする。
	- `//`:  `--`の後の空白がWebアプリのバリデーションによって切り捨てられないようにする。
- `OR 1=1 -- //`を挿入する方法。常にTrueとなる。以降の条件をコメントとする。
#### 具体例

- URLが以下
```
https://insecure-website.com/products?category=Gifts
```
- それによるSQLが以下で、かつ、SQLiの脆弱性対策をしていないとする
```sql
SELECT * FROM products WHERE category = 'Gifts' AND released = 1
```

- `-- //`を挿入。`'`はGiftsというStringを囲うため
```
https://insecure-website.com/products?category=Gifts'--
```
- それによるSQLが以下
```sql
SELECT * FROM products WHERE category = 'Gifts'--' AND released = 1
```

- `OR 1=1--`を挿入。これにより、category='Gifts'という条件がバイパスされる
```
https://insecure-website.com/products?category=Gifts'+OR+1=1--
```
```sql
SELECT * FROM products WHERE category = 'Gifts' OR 1=1--' AND released = 1
```

- ⚠️: OR 1=1がUPDATEやDELETEで使用されると全てのデータが変更され危険なので慎重に

## アプリケーションロジックの破壊

- 例：ログインフォーム
- username=wiener, password=peterと入力したSQL
```sql
SELECT * FROM users WHERE username = 'wiener' AND password = 'peter'
```

- パスワード入力なしOKの場合
```sql
SELECT * FROM users WHERE username = 'administrator'--' AND password = ''
```

- パスワードの入力が必要な場合
```sql
SELECT * FROM users WHERE username = 'administrator' AND password = 'anything' OR 1=1--
```
```sql
SELECT * FROM users WHERE username = 'administrator'--' AND password = 'anything'
```

## SQLiの基本的な文法

###### サブクエリ

- 以下のようにするのは、SQLiの脆弱性がある箇所がWHERE句の一部で、注入したいクエリをANDにつづく条件として記載するため、AND()のように括弧が必要
```sql
xyz' AND (SELECT ~)
```

- サブクエリの考え([参考記事](https://products.sint.co.jp/siob/blog/what-is-sql-subquery))
```sql
SELECT *  
FROM EMPLOYEES  
WHERE DEP_NAME = '営業' AND (SELECT ID FROM DEPARTMENT WHERE NAME = '開発部')
```

###### ヘルスチェック

- SELECT句で==存在しないカラム==を指定して、正常に動作しているか確認できる
```sql
SELECT [NULL | 1] FROM table
```
- NULLと1は軽量。NULLが便利だが、DBの構造によっては1を使用する。

###### 注入の仕方

- 元々の結果のなかに抽出したいデータを表示する
```sql
' AND (SELECT ~)
```

- 元々の結果と関わりなく意図する動作を引き起こす(Time-based SQLiなど)
```sql
'; SELECT ~
```
- `;`で元々のクエリを終了させる

## SQLiでの推測方法

###### 推測

- Webアプリケーションに表示されているデータから、カラムのデータ型をある程度は推測できる。例えば、アプリケーションが以下のような形式だとする
```
1
Hi, My name is Mike.
```
- この場合、カラム1がint型、カラム2がvarchar型など想像できる。

###### エラーが起きた時の考え方の展開

- `ORDER BY 1--`でエラー　：　コメントアウトが--ではなく#かも。

###### 重要そうなものを見極める

- バックエンドシステムと通信しているであろうリクエストはRepeaterへ送り操作する。例えばPOSTやGETのクエリストリングに値を付与しているときなど

## 🛡️SQLiの対策

- プレースホルダーでパラメータ(値）化されたクエリがSQLiの防止に有効であるためには、クエリで使用する文字列は必ずハードコードされた定数でなければならない。変数などを使用しない。
- データ項目が信頼できるかどうかをケースバイケースで判断は🙅‍♀️。他のコードの変更によって信頼できるデータが汚染されたりしてSQLiの脆弱性となることがある

###### プリペアドステートメントを使用

- ↓脆弱な例
```sql
String query = "SELECT * FROM products WHERE category = '"+ input + "'"; Statement statement = connection.createStatement();
ResultSet resultSet = statement.executeQuery(query);
```
- 外部からの入力（`input`変数）を直接クエリに挿入している

- これをプリペアドステートメントに書き換えたもの
```sql
PreparedStatement statement = connection.prepareStatement("SELECT * FROM products WHERE category = ?"); statement.setString(1, input); 
ResultSet resultSet = statement.executeQuery();
```
- `?`がプレースホルダー。外部からの入力がクエリの一部としてではなく、値として扱われる
- [詳細な参考記事](https://www.javadrive.jp/servlet/database/index10.html)

###### プリペアドステートメントが使えないとき

- プリペアドステートメントはtable・column名やORDER BY句では使用不可なので他の方法を取る
	- 入力値のホワイトリストを作成する
	- 同様の動きをする別のロジックを考える
