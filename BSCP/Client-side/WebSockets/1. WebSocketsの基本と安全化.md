## HTTPとの違い

###### リアルタイム性に優れている

- クライアントとサーバの双方向で通信できる
- コネクションを一度確立すると、そのコネクションを使って通信を行う
- サーバ側から状態に変化があるたびにリクエストを送ることができる
- 例えば、刻一刻と変化する相場情報など

###### HTTPのデメリット

- クライアントからしかリクエストできない(サーバからの通信ができない)＝双方向通信は不可能
- 1つのコネクションで1つのリクエストしか送れないので通信効率が悪い
- 刻一刻と変化する情報を取得するためにクライアントから頻繁にリクエストを送り、帯域を圧迫

## WebSocketコネクションの確立の流れ


1. クライアントからサーバへWebSocketに対応しているか確認する
2. クライアントから`upgrade`ヘッダを含むGETリクエストを送信してハンドシェイクする

- 以下のようなハンドシェイクリクエストになる
```http
GET /chat HTTP/1.1
Host: normal-website.com
Sec-WebSocket-Version: 13
Sec-WebSocket-Key: wDqumtseNBJdhkihL6PW7w==
Connection: keep-alive, Upgrade
Cookie: session=...
Upgrade: websocket
```
- `upgrade`:　プロトコルの変更を行う

3. サーバがクライアントからのWebSocket変更を了解すると以下のようなレスポンスを返す
```http
HTTP/1.1 101 Switching Protocols
Connection: Upgrade
Upgrade: websocket
Sec-WebSocket-Accept: 0FFP+2nmNIf/h+4BP36k9uzrYGk=
```

###### WebSocket handshakeの特徴

- リクエストとレスポンスにある`Connection`と`Upgrade`ヘッダはWebSocket handshakeを示している
- `Sec-WebSocket-Version`: クライアントが使用したいWebSocket プロトコルバージョン(通常13)
- `Sec-WebSocket-Key`: Base64エンコードされたランダム値を含む
- レスポンスの`Sec-WebSocket-Accept`：リクエストの`Sec-WebSocket-Key`で送信された値をハッシュ化し、特定の文字列と連結する。これはサーバの設定ミスやプロキシのキャッシュによるエラーを防ぐ

## WebSocketメッセージの見た目

- 通常はJavaScriptでインスタント化する
```js
var ws = new WebSocket("wss://normal-website.com/chat");
```
- `wss`: WebSocketのTLS接続。暗号化しない場合は`ws`となる

- メッセージ例
```js
ws.send("Peter Wiener");
```

- WebSocketメッセージは任意のコンテンツやデータ形式を含むことができる(JSONが一般的)
```js
{"user":"Kosi ikeda","content":"こんにちは！"}
```

## WebSocketの安全化

- `wss://`を使用：TLS接続
- WebSocketsエンドポイントのURLをハードコードし、このURLにユーザーが制御可能なデータを組まない
- WebSocketハンドシェイクメッセージをCSRFから保護し、クロスサイトWebSocketハイジャックの脆弱性を回避する
- WebSocket経由で受信したデータは、双方向で信頼できないものとして扱う。SQLiやXSSのような入力ベースの脆弱性を防ぐために、サーバ側とクライアント側の両方でデータを安全に扱う。
